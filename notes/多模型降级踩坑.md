# 多模型降级踩坑笔记

> `llm_service` 的 Provider 抽象 + fallback 是设计文档里的「多模型容灾」。做的时候才发现「切个备用模型」远没想的简单。全是后端 LLM 调用与容灾的坑。

---

## 坑 1：业务层直连 SDK，换模型要改一堆地方

**现象**：一开始图快，`chat_service` 里直接 `openai.chat.completions.create(...)`。后来要加 Claude 备用，发现调用代码散在好几处，每处都要改，还各自处理一遍异常。

**根因**：没做 Provider 抽象，业务层和具体供应商 SDK 耦死了。

**解决**：抽一个 `LLMProvider` 接口，每个供应商实现它，业务层只认接口：

```python
class LLMProvider(Protocol):
    async def chat(self, messages, **kw) -> LLMResult: ...
    async def stream(self, messages, **kw) -> AsyncIterator[str]: ...

class OpenAIProvider(LLMProvider): ...
class ClaudeProvider(LLMProvider): ...

# llm_service 持有 provider 列表，业务层只调 llm_service.chat()
```

返回值也统一成自己的 `LLMResult`（含 text、prompt_tokens、completion_tokens、model_name），不同 SDK 的响应结构在 Provider 内部就抹平。

**教训**：外部 LLM/embedding/reranker 全部经 service 抽象（设计文档 ch.11 明说了）。先抽接口再接第二家，比事后重构省太多事。

---

## 坑 2：超时没设，主 Provider 卡住整个请求一起卡死

**现象**：主 Provider 偶发响应特别慢（几十秒），用户请求就一直挂着，fallback 根本没机会触发。

**根因**：没设调用超时。fallback 是「失败了才切」，但「一直不返回」既不算成功也不算失败，卡在那不动，容灾形同虚设。

**解决**：每次调用设超时，超时当失败处理触发 fallback：

```python
try:
    return await asyncio.wait_for(provider.chat(messages), timeout=LLM_TIMEOUT)
except (asyncio.TimeoutError, ProviderError):
    # 落入 fallback 链
    ...
```

**教训**：容灾的前提是「能及时判定失败」。超时是把「慢」转成「失败」的关键，不设超时的 fallback 是假容灾。

---

## 坑 3：fallback 链没有上限，备用也挂了就无限重试

**现象**：某次主备同时抖动，请求在 fallback 链里反复重试，迟迟不返回。

**根因**：fallback 写成了「失败就换下一个」，但没限制总尝试次数，也没区分「可重试错误」和「不可重试错误」（比如 4xx 参数错误重试多少次都没用）。

**解决**：fallback 链固定顺序 + 总次数上限 + 错误分类，全挂了就明确降级返回：

```python
last_err = None
for provider in self.fallback_chain:          # 有限的、有序的链
    try:
        if is_non_retryable(last_err):        # 参数类错误不重试，直接抛
            raise last_err
        return await self._call_with_timeout(provider, messages)
    except ProviderError as e:
        last_err = e
        log_switch(provider, e)               # 记录切换事件
        continue
# 全部失败：明确降级，别让用户干等
raise AllProvidersFailed(last_err)
```

问答侧捕获 `AllProvidersFailed`，返回友好提示而不是 500。

**教训**：fallback 链要「有限 + 有序 + 错误分类」。429/超时/5xx 可切下一个，4xx 参数错误别浪费重试。

## 坑 4：限速（429）没感知，一直往同一个限速的 key 上撞

**现象**：高频请求时频繁 429。fallback 虽然会切，但下次请求又先撞主 key 的 429，再切，每次都白撞一下，延迟和浪费都高。

**根因**：fallback 是被动的「撞墙才换」，没主动感知哪个 key 还有余量。同一个 key 限速窗口没过，下次照样先撞它。

**解决**（设计文档 ch.18 「限速感知调度」的雏形）：用 Redis 计数跟踪各 key 的 RPM 余量，命中 429 后把该 key 短暂「冷却」，优先发给余量足的：

```python
if redis.get(f"cooldown:{key}"):
    continue                       # 这个 key 还在冷却，跳过
try:
    return await provider.chat(...)
except RateLimitError as e:
    redis.setex(f"cooldown:{key}", retry_after, 1)   # 按 Retry-After 冷却
    continue
```

命中 429 时尊重响应里的 `Retry-After` 做退避，别立刻重撞。

**教训**：限速要主动感知 + 退避，不能光靠 fallback 被动硬撞。Redis 计数 + 冷却是 MVP 阶段够用的轻量做法，再往上就是 ch.18 的负载均衡了。

## 坑 5：流式中途切 Provider，token 流断成两截

**现象**：流式问答时主 Provider 生成到一半挂了，我让它 fallback 到备用接着生成，结果前半段和后半段是两个模型生成的，语义接不上，token 统计也乱了。

**根因**：流式连接已经开始吐 token 了，中途换模型等于「换了个人接着写」，上下文和语气都对不上，没法无缝拼接。

**解决**（按设计文档 ch.18 「流式兼容」原则）：fallback 只在 **建立连接阶段** 生效，一旦开始 yield token 就锁定在当前 Provider，中途失败就结束这次流并落库已生成部分，不中途切：

```python
provider = await select_provider_with_fallback()   # 建连阶段可 fallback
started = False
try:
    async for delta in provider.stream(messages):
        started = True
        yield format_sse("token", {"text": delta})
except ProviderError:
    if not started:
        provider = next_fallback()        # 还没吐 token，可以换
        # 重新开流
    else:
        # 已经吐了一半，不换，收尾落库已生成部分
        finalize_partial()
```

**教训**：流式和非流式的容灾策略不一样。非流式可以随便重试切换；流式一旦开吐就不能换人，只能「建连阶段挑好，中途坏了就优雅收尾」。
